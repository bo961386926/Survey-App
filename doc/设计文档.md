# 青泓项目勘察信息采集与审核系统 - 设计文档

## 版本信息

- 当前版本：V5.0 汇总版
- 更新日期：2026-05-29
- 来源合并：UI 设计语言、OpenAPI 草案、前端路由与页面拆分、技术方案审查、开发任务清单、部署质量保障方案、后端完成情况报告

## 1. 总体架构

### 1.1 技术栈

- PC 后台：Vue3、Vue Router、Pinia、Ant Design Vue / Element Plus 兼容现有 Soybean Admin 结构。
- 移动端：uni-app、Pinia、uView Plus。
- 服务端：Spring Boot、Spring Security、JWT、MyBatis Plus、RESTful API。
- 数据库：MySQL 8.0。
- 缓存与会话辅助：Redis。
- 文件存储：OSS。
- 地图服务：高德地图、微信小程序原生 map。
- 文档与调试：OpenAPI/Swagger/Knife4j。

### 1.2 模块划分

- `admin-backend`：后端服务、认证鉴权、业务接口、日志、导出、文件上传。
- `admin-web-soybean`：PC 后台管理端。
- `mobile-app`：移动采集端。
- `init-sql` / `admin-backend/init-data`：数据库初始化和迁移脚本。
- `doc`：当前保留需求、设计、测试三份最新文档。

### 1.3 核心领域对象

- 项目：`project`
- 标段：`project_section`
- 项目成员：`project_member`
- 点位：`survey_point`
- 模板：`survey_template`
- 模板版本：`survey_template_version`
- 采集结果：`survey_result`
- 审核记录：`survey_audit_record`
- 导出任务：`export_task`
- 协作入口：`collab_entry`
- 操作日志：`operation_log`
- 纠偏日志：`location_correction_log`

## 2. 数据设计

### 2.1 关键表变更

- `survey_point` 增加 `outfall_type varchar(50) not null`，表示场景类型。
- `survey_result` 必须包含 `version_no int not null`，按点位递增。
- `survey_audit_record.action` 支持 `pass`、`reject`、`transfer`，其中 `transfer` 本期预留。
- `collab_entry.token` 用于协作访问入口，不与主系统登录 Token 混用。
- 新增 `location_correction_log` 记录纠偏前后坐标、用户、点位、采集结果和时间。

### 2.2 状态设计

- 点位展示状态由最新采集结果聚合生成。
- 采集结果是真实业务状态与版本承载表。
- 审核通过后可触发 PDF 生成和归档。
- 驳回后复制原数据生成新草稿版本，避免原历史被覆盖。

### 2.3 并发设计

- 点位/采集结果编辑采用数据版本号或更新时间校验。
- 客户端打开记录时获取版本号。
- 保存或提交时提交版本号。
- 服务端发现版本不一致时拒绝写入，并返回“数据已被他人修改，请刷新后重试”。

## 3. 接口设计

### 3.1 通用约定

- Base URL：`/api`
- 请求头：`Authorization: Bearer {token}`、`Content-Type: application/json`
- 通用返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "trace-001"
}
```

- 分页结构：

```json
{
  "list": [],
  "pageNo": 1,
  "pageSize": 20,
  "total": 100
}
```

### 3.2 接口分组

- 认证：`POST /auth/login`、`POST /auth/sms-login`、`POST /auth/logout`、`POST /auth/reset-password`、`GET /auth/profile`
- 项目：`GET /project/list`、`POST /project/create`、`PUT /project/update`、`GET /project/detail`、`POST /project/archive`
- 标段与成员：`GET /project/section/list`、`POST /project/section/create`、`PUT /project/section/update`、`GET /project/member/list`、`POST /project/member/bind`
- 模板：`GET /template/list`、`POST /template/create`、`PUT /template/update-draft`、`POST /template/publish`、`GET /template/version/list`、`GET /template/detail`、`POST /template/bind-project`
- 点位：`GET /point/list`、`POST /point/import`、`POST /point/create`、`PUT /point/update`、`POST /point/assign`、`GET /point/detail`、`GET /point/map`、`POST /point/discard`
- 采集：`GET /survey/detail`、`POST /survey/save-draft`、`POST /survey/submit`、`GET /survey/version/list`、`GET /survey/version/detail`
- 审核：`GET /audit/todo-list`、`GET /audit/detail`、`POST /audit/pass`、`POST /audit/reject`、`POST /audit/batch-pass`
- 导出：`POST /export/create`、`GET /export/list`、`GET /export/detail`、`POST /export/regenerate-pdf`
- 协作：`POST /collab/create`、`POST /collab/revoke`、`POST /collab/reset-token`、`GET /collab/access-log`
- 系统：用户、角色、权限、字典、消息、操作日志。

### 3.3 重点接口规则

- 登录成功返回 `token`、`refreshToken`、`expiresIn`、`userInfo`。
- 点位列表和地图接口必须支持项目、标段、状态、场景类型、关键字筛选。
- 采集详情返回点位信息、模板版本、已有草稿或最新结果、数据版本号。
- 保存草稿和提交审核必须校验数据版本号。
- 审核接口必须校验是否最新版本。
- 协作 Token 调用接口时必须校验 `login_type` 与授权范围。

## 5. UI 设计规范

### 5.1 视觉方向

- 设计风格：冰蓝极简、Apple 简约美学。
- 核心材质：毛玻璃，卡片、面板、导航栏使用半透明白和背景模糊。
- 背景：`linear-gradient(180deg, #F0F7FF 0%, #FFFFFF 100%)`。
- 页面容器/大卡片圆角：`24px`。
- 按钮/图标容器圆角：`12px` - `16px`。
- 阴影：`0 8px 32px rgba(0, 40, 80, 0.05)`。

### 5.2 色彩

- 品牌主色：`#1664FF`
- 辅助冰蓝：`#7DD3FC`
- 一级文本：`#0F172A`
- 正文文本：`#475569`
- 辅助文本：`#94A3B8`
- 进行中：`#3B82F6`
- 已驳回/逾期：`#EF4444`
- 已完成/通过：`#10B981`

### 5.3 字体

- 首选字体：`Manrope` + `PingFang SC`
- TopAppBar 标题：18px Semi-Bold，居中。
- 卡片主标题：16px Semi-Bold。
- 表单标签/正文：14px Regular。
- 辅助文字：12px Medium。

### 5.4 组件原则

- 地图、表格、审核详情、模板设计器优先保证信息密度与操作效率。
- 移动端表单控件必须适合单手操作，错误反馈就近展示。
- 状态标签颜色与点位地图颜色保持一致。
- 空状态用于无任务、无消息、无草稿等场景，必须提供下一步入口。
- 二级页面统一返回逻辑，顶部导航保持毛玻璃效果。

## 6. 后端设计要点

### 6.1 安全

> **详细权限体系设计见 [权限体系设计.md](./权限体系设计.md)，可视化全景见 [权限体系设计-可视化.html](./权限体系设计-可视化.html)。**

- Spring Security + JWT 鉴权（双通道：内部用户 / 协作入口）。
- RBAC 五表模型（`sys_user` → `sys_user_role` → `sys_role` → `sys_role_permission` → `sys_permission`）。
- 权限校验覆盖 Controller（`@PreAuthorize` 注解）与核心 Service。
- 前端双层权限控制：路由守卫（`meta.roles`）+ UI 权限（`hasPermission` / `v-permission`）。
- 数据级隔离：`project_member` 项目成员范围 + `collab_entry` 第三方协作范围。
- 第三方协作采用白名单+黑名单双重策略（`CollabSecurityService`）。
- 权限码自动注册与同步（`PermissionRegistry` + `PermissionInitializer`）。
- 通配符 `*` 权限展开为超级管理员全部权限。
- 全链路操作审计：`@OperationLog` 注解 + AOP 切面异步记录。
- 禁用账号、角色变更、Token 过期需及时生效。
- 文件下载、导出下载必须鉴权。

### 6.2 日志

- 登录、审核、导出、权限变更、协作访问、风控拦截、纠偏操作均需留痕。
- 日志需支持筛选、风险分级、导出和链路查看。

### 6.3 导出

- 小数据可同步生成。
- 大批量导出和批量 PDF 必须异步生成 `export_task`。
- 导出完成发送消息提醒。
- 下载链接可设置过期时间。

### 6.4 文件与图片

- 图片原图上传 OSS。
- 展示时拼接动态水印。
- 移动端根据模板 Image 字段决定是否只允许相机拍摄。
- 图片上传失败支持单张重试。

## 7. 部署与质量保障设计

- 后端暴露健康检查接口。
- Swagger/Knife4j 必须可用于联调和契约检查。
- 部署前执行后端测试、前端类型检查、关键接口探活。
- API 变更需更新 OpenAPI 文档，并评估前端调用影响。
- 建议 CI/CD 阶段加入接口 diff、单元测试、集成测试、前端构建和 Docker 镜像构建。

## 8. 开发优先级

- P0：认证与权限、项目管理、点位管理、动态模板、移动端采集、草稿/提交、基础审核。
- P1：地图中心、导出报表、第三方协作、批量审核、消息提醒。
- P2：高级日志分析、SaaS 租户隔离、更多统计看板、转审前端能力。
