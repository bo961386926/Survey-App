# 部署质量保障方案

## 问题背景
部署后登录失败，原因是前后端接口契约不一致：
- 后端接口路径: `/api/auth/login`
- 前端调用路径: `/auth/login` (通过代理转换)
- 类型定义手动维护，容易出现不一致

## 优化方案

### 1. API 契约管理 (OpenAPI/Swagger)

**后端已有 Swagger，需要启用并导出 OpenAPI 规范：**

```yaml
# application.yml 添加
springdoc:
  api-docs:
    enabled: true
    path: /v3/api-docs
  swagger-ui:
    enabled: true
    path: /swagger-ui.html
```

**前端自动生成类型：**

```bash
# 安装 openapi-typescript
pnpm add -D openapi-typescript

# 生成类型（在 package.json scripts 中添加）
"scripts": {
  "gen:api-types": "openapi-typescript http://localhost:8080/v3/api-docs --output src/typings/api-generated.d.ts"
}
```

### 2. 集成测试 - 登录流程

**创建后端集成测试：**

```java
// AuthControllerIntegrationTest.java
@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
class AuthControllerIntegrationTest {
    
    @Test
    void testLoginFlow() {
        // 1. 获取验证码
        var captchaResp = mockMvc.perform(get("/api/auth/captcha"))
            .andExpect(status().isOk())
            .andReturn();
        
        var captchaData = parseResponse(captchaResp);
        String key = captchaData.get("key");
        
        // 2. 登录
        mockMvc.perform(post("/api/auth/login")
            .contentType(MediaType.APPLICATION_JSON)
            .content("""
                {
                    "username": "admin",
                    "password": "admin123",
                    "captchaKey": "%s",
                    "captcha": "%s"
                }
                """.formatted(key, captchaData.get("code"))))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.data.accessToken").exists());
        
        // 3. 获取用户信息
        mockMvc.perform(get("/api/auth/getUserInfo")
            .header("Authorization", "Bearer " + token))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.data.userName").value("admin"));
    }
}
```

### 3. E2E 测试 (Playwright)

**前端 E2E 测试：**

```typescript
// tests/login.spec.ts
import { test, expect } from '@playwright/test';

test('登录流程', async ({ page }) => {
  await page.goto('/login');
  
  // 等待验证码加载
  await expect(page.locator('.captcha-image')).toBeVisible();
  
  // 获取验证码（测试环境可使用固定验证码）
  const captcha = await page.locator('.captcha-image').getAttribute('data-code');
  
  // 输入登录信息
  await page.fill('[name="username"]', 'admin');
  await page.fill('[name="password"]', 'admin123');
  await page.fill('[name="captcha"]', captcha || '1234');
  
  // 点击登录
  await page.click('button[type="submit"]');
  
  // 验证登录成功
  await expect(page).toHaveURL(/\/home/);
});
```

### 4. 部署前检查清单

**创建部署检查脚本：**

```bash
#!/bin/bash
# scripts/pre-deploy-check.sh

echo "=== 部署前检查 ==="

# 1. 检查后端健康状态
echo "1. 检查后端服务..."
curl -f http://localhost:8080/actuator/health || exit 1

# 2. 检查 Swagger 文档
echo "2. 检查 API 文档..."
curl -f http://localhost:8080/v3/api-docs || exit 1

# 3. 运行后端测试
echo "3. 运行后端测试..."
cd admin-backend && mvn test || exit 1

# 4. 运行前端类型检查
echo "4. 检查前端类型..."
cd admin-web-soybean && pnpm type-check || exit 1

# 5. 检查关键接口
echo "5. 检查关键接口..."
curl -f http://localhost:8080/api/auth/captcha || exit 1

echo "=== 所有检查通过 ==="
```

### 5. CI/CD 流程优化

**GitHub Actions 工作流：**

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: 后端测试
        run: cd admin-backend && mvn test
      
      - name: 前端类型检查
        run: cd admin-web-soybean && pnpm type-check
      
      - name: E2E 测试
        run: |
          # 启动服务
          docker-compose up -d
          # 等待服务就绪
          sleep 30
          # 运行 E2E 测试
          cd admin-web-soybean && pnpm test:e2e

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: 构建并部署
        run: |
          docker-compose build --no-cache
          docker-compose up -d
```

### 6. 接口变更管理规范

**规则：**
1. 后端接口变更必须更新 Swagger 文档
2. 前端类型必须从 OpenAPI 自动生成
3. 重大变更需要通知前端开发者
4. 部署前必须运行集成测试

**接口变更检查脚本：**

```bash
#!/bin/bash
# scripts/api-diff-check.sh

# 对比新旧 API 规范
old_api="docs/api-spec-old.json"
new_api="http://localhost:8080/v3/api-docs"

# 下载新规范
curl -s $new_api > docs/api-spec-new.json

# 对比差异
diff -u $old_api docs/api-spec-new.json > docs/api-diff.txt

if [ -s docs/api-diff.txt ]; then
    echo "警告：API 规范有变更！"
    cat docs/api-diff.txt
    echo "请确认前端已同步更新"
    exit 1
fi
```
